## Abstract

ZBX Chain implements ERC-4337 Account Abstraction natively with three
extensions: **Session Keys** (time/scope-limited signing keys for dapps),
**Social Recovery** (recover lost keys via trusted guardians), and
**Gas Sponsorship** (dapps pay user gas). These features make smart
contract wallets first-class citizens on ZBX Chain.

---

## Motivation

EOA (Externally Owned Account) wallets have fundamental UX limitations:
- Lost private key = lost funds forever (no recovery)
- Users must hold native token to pay gas (friction for new users)
- Every dapp interaction requires wallet popup (bad UX for games/social)
- No spending limits or per-dapp permissions

Account Abstraction replaces EOA restrictions with programmable validation logic.

## Specification

### 1. Core ERC-4337 (EntryPoint Contract)

EntryPoint deployed at: `0x5FF137D4b0FDCD49DcA30c7CF57E578a026d2789`

```solidity
struct UserOperation {
    address sender;           // smart contract wallet address
    uint256 nonce;
    bytes   initCode;         // wallet factory calldata (first tx only)
    bytes   callData;         // wallet.execute() calldata
    uint256 callGasLimit;
    uint256 verificationGasLimit;
    uint256 preVerificationGas;
    uint256 maxFeePerGas;
    uint256 maxPriorityFeePerGas;
    bytes   paymasterAndData; // optional paymaster
    bytes   signature;        // wallet-specific validation
}
```

### 2. Session Keys (ZBX Extension)

Session keys allow dapps to sign transactions on behalf of users within
defined constraints — no wallet popup needed for every action.

```solidity
struct SessionKeyPolicy {
    address sessionKey;       // ephemeral signing key (generated by dapp)
    address targetContract;   // dapp contract only (scope limit)
    bytes4  allowedSelector;  // specific function only (e.g. "play()")
    uint256 maxValuePerTx;    // max ZBX value per transaction
    uint256 dailyLimit;       // total ZBX per day
    uint256 validFrom;        // unix timestamp
    uint256 validUntil;       // unix timestamp (max 7 days)
}
```

Wallet validates session key usage against registered `SessionKeyPolicy`.
If all constraints pass → no user signature required.

**Use cases**: GameFi (auto-move), SocialFi (auto-like/post), DEX bots (auto-rebalance)

### 3. Social Recovery (ZBX Extension)

```solidity
struct RecoveryConfig {
    address[] guardians;      // trusted friends/family/devices
    uint256   threshold;      // guardians needed to recover (e.g. 3-of-5)
    uint256   recoveryDelay;  // timelock before recovery executes (e.g. 48h)
}
```

Recovery flow:
```
1. User loses key
2. threshold guardians initiate recovery (each calls initiateRecovery())
3. recoveryDelay passes (gives owner time to cancel if not really lost)
4. Any guardian executes recovery → new owner key set
```

Guardians can be: other EOAs, hardware wallets, email recovery services (ZK-based),
or even a ZBX DAO multisig.

### 4. Gas Sponsorship (Paymaster)

```solidity
interface IPaymaster {
    function validatePaymasterUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 maxCost
    ) external returns (bytes memory context, uint256 validationData);

    function postOp(
        PostOpMode mode,
        bytes calldata context,
        uint256 actualGasCost
    ) external;
}
```

**ZBX Paymaster types**:
- `FreeGasPaymaster` — dapp pays all gas (user onboarding)
- `ERC20Paymaster` — user pays gas in ZUSD/ZNS (no ZBX needed)
- `SubscriptionPaymaster` — monthly subscription → unlimited gas

### 5. Bundler Protocol

Bundlers collect UserOperations and submit them as bundle transactions:

```
UserOp → bundler mempool → simulation → bundle tx → EntryPoint.handleOps()
```

ZBX native bundler (`zbx-bundler` crate) exposes:
- `eth_sendUserOperation`
- `eth_estimateUserOperationGas`
- `eth_getUserOperationReceipt`
- `eth_supportedEntryPoints`

### 6. Native Protocol-Level Integration (ZBX-specific)

Unlike Ethereum (where ERC-4337 is fully contract-based), ZBX Chain integrates
AA at protocol level for efficiency:

- **Mempool-level validation**: bundler mempool rejects invalid UserOps early
- **Fee market**: UserOps participate in the same EIP-1559 fee market
- **Gas accounting**: preVerificationGas calibrated to ZBX block parameters
- **ZBX gas token support**: UserOps can specify `gas_token` (ZBX or ZUSD)

## Security

- **Session key scope**: contract + selector + value + time — all checked on-chain
- **Recovery timelock**: 48h window lets owner cancel false recovery
- **Guardian privacy**: guardian addresses are hashed in storage (anti-social-engineering)
- **Replay protection**: UserOp nonce is strictly sequential per sender
- **Simulation**: all UserOps simulation-verified before mempool inclusion
